Counting Method

EPT counts structured diagnostics, not text. The predicate is:

defp type_warning?(%{severity: :warning, details: %{typing_traces: _}}), do: true
defp type_warning?(_), do: false

The counter runs after the measured mix compile --force --profile time finishes. It opens the project through Mix and reads the Elixir compiler diagnostics stored by the compile manifest. For umbrella projects it recursively enters each child app and reads each child's Elixir diagnostics.

case Mix.Project.apps_paths() do
  nil -> Mix.Tasks.Compile.Elixir.diagnostics()
  apps ->
    Enum.flat_map(apps, fn {app, path} ->
      Mix.Project.in_project(app, path, fn _ -> diagnostics_for_project() end)
    end)
end

Why this catches all type warnings

In this Elixir checkout, every type-checker warning path that needs the special type-warning rendering attaches details: %{typing_traces: traces}. This appears across the type checker modules under Module.Types: apply.ex, expr.ex, of.ex, and pattern.ex.

The user-visible marker in the compiler log is produced by the same structured field:

format_detail(#{details := #{typing_traces := _}}, Message) ->
  [Message | "\ntype warning found at:"];

Therefore, if a warning prints as a type warning in the log, it had details.typing_traces. Conversely, if a warning lacks that field, the formatter does not print type warning found at:, and EPT does not count it.

Grouped diagnostics are also handled. Module.ParallelChecker.emit_diagnostics/3 expands each grouped diagnostic into one diagnostic per location before returning it to Mix. The audit counted both the marker blocks and the final location lines, and they matched EPT's structured count for every repo.

Why dependencies are excluded

perf.py compiles dependencies in a separate setup step with mix deps.compile --force. The measured project compile then runs from the project root with mix compile --force --no-deps-check. The type-warning counter is invoked only after that project compile and reads diagnostics from that project's compile manifests. Dependency warning text may appear during setup, but it is not included in the counted project section.

Failure handling

If compilation times out or fails before diagnostics are reliable, EPT reports a status such as compile failed, timeout, or not compiled rather than inventing a count. If serious-run mode produces inconsistent warning counts between measured runs, the table reports the inconsistent list instead of a single count.
